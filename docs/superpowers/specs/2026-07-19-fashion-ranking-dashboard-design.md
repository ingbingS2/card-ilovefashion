# 설계 — 패션 랭킹 대시보드 + 원클릭 카드뉴스 파이프라인

- 작성일: 2026-07-19
- 상태: 사용자 승인됨 (A안: 조회는 클라우드, 제작·업로드는 로컬 원클릭)

## 목적

무신사·29CM의 카테고리별 랭킹과 상품별 별점·후기를 한 화면에서 통합 비교하는 웹사이트를 만들고,
사이트에서 사용자가 직접 선택한 상품들만으로 카드뉴스를 제작해 인스타그램에 게시하는 것까지
버튼 한 번(게시 직전 미리보기 확인 1회 포함)으로 잇는다.

## 확정된 요구사항

| 항목 | 결정 |
|---|---|
| 사용자 | 주로 본인, 링크 공유 시 지인도 조회 가능 (조회는 공개, 조작은 본인만) |
| 데이터 갱신 | GitHub Actions가 1시간 단위 크롤링 → DB 저장, 사이트는 DB만 읽음 |
| 수집 범위 | 주요 패션 카테고리(상의·바지·스커트/원피스·가방·신발·액세서리, 설정으로 조정) × TOP 50 × 무신사+29CM |
| 후기 | 평점·후기 수는 매시간, 후기 본문은 상품당 10개를 증분 수집(후기 수 변동·신규 진입 상품만) |
| 화면 | 두 몰 통합 비교(카테고리 탭 → 무신사·29CM 2열 랭킹), 상품 클릭 시 후기 상세 |
| 파이프라인 | 사이트에서 체크한 상품만 → 문구 생성 → C안 템플릿 렌더 → 미리보기 확인 → 인스타 캐러셀 게시 |
| 게시 검수 | 완성 JPG 미리보기 후 "게시" 버튼 1회 클릭 (완전 무인 아님) |

## 전체 구조 (기존 저장소 확장)

```
crawler/           # [신규] Python 크롤러 — GitHub Actions에서 매시간 실행
frontend/          # [개조] 기존 React+Vite+TS 앱을 랭킹 대시보드로 확장
pipeline/          # [신규] 로컬 원클릭 앱(FastAPI) — 문구 생성→렌더→업로드
scripts/           # [기존] upload_instagram.py 로직 재사용
card-drafts/       # [기존] C안(인스타 후킹형) 템플릿 → 렌더 템플릿으로 재사용
```

- Firestore가 크롤러(쓰기)와 사이트(읽기) 사이의 접점.
- 기존 Firebase Hosting, GitHub Actions 구조를 그대로 확장.
- 모든 사용자 노출 UI는 한국어 (CLAUDE.md 규칙).

## 컴포넌트 상세

### 1) crawler/ — 데이터 수집

- 무신사·29CM 웹 프론트가 사용하는 JSON API를 `requests`로 호출 (HTML 파싱 대비 안정적).
- 매시간 작업: 카테고리 × 2몰 → 순위, 상품ID, 브랜드, 상품명, 가격, 할인율, 평점 평균,
  후기 수, 썸네일 URL, 상품 페이지 URL.
- 후기 본문 증분 수집: 직전 스냅샷과 비교해 (a) 후기 수가 변했거나 (b) 랭킹에 새로 진입한
  상품만 상품당 최신·베스트 10개 수집. 요청 간 딜레이와 재시도(지수 백오프) 내장.
- 카테고리 목록은 `crawler/config.yaml`(또는 동급 설정 파일)로 관리 — 코드 수정 없이 추가/제거.
- 스케줄: `.github/workflows/crawl.yml` — `schedule: cron('0 * * * *')` + 수동 트리거(workflow_dispatch).
- Firestore 쓰기는 서비스 계정(GitHub Secrets `FIREBASE_SERVICE_ACCOUNT`) 사용.

#### Phase 0 — 선행 검증 (본 구현 전 필수)

1회용 workflow_dispatch 워크플로로 GitHub Actions 러너에서 두 몰의 랭킹·후기 API가
실제로 응답하는지 확인한다. 차단 시 크롤러 실행 위치만 "내 PC 작업 스케줄러"로 전환하고
나머지 설계(코드·DB·사이트)는 그대로 유지한다.

### 2) Firestore 스키마

| 컬렉션/문서 | 내용 |
|---|---|
| `rankings/{mall}_{category}` | 최신 스냅샷: TOP 50 배열(경량 상품 필드), `updatedAt`. 사이트 목록 화면은 이 문서만 읽음 |
| `products/{mall}_{productId}` | 상품 상세 + 후기 최대 10개 + 시계열 이력(시각, 순위, 평점, 후기 수) — 순위 변동(▲▼) 표시용 |
| `selections/{yyyymmdd-hhmm}` | 사이트에서 체크한 상품 참조 목록 + 상태(`pending`/`processing`/`done`/`failed`) — 파이프라인 입력 |

- 보안 규칙: `rankings`·`products` 읽기 공개 / 쓰기 전면 차단(서비스 계정=Admin SDK는 규칙 미적용).
  `selections` 쓰기는 파이프라인·본인만 필요하므로 규칙상 쓰기 차단하고 로컬 앱(Admin SDK)을 통해 기록.
- 사이트의 선택 버튼은 로컬 파이프라인 앱을 경유해 `selections`에 기록한다(아래 4장 흐름 참고).
  따라서 지인은 조회만 가능하고 선택/게시는 구조적으로 불가능.

### 3) frontend/ — 조회 사이트

- 카테고리 탭 → 무신사·29CM 나란히 2열 랭킹 리스트 (통합 비교).
- 상품 카드: 순위, 순위 변동(▲▼, 이력 기반), 썸네일, 브랜드, 상품명, 가격/할인율, ⭐평점, 후기 수.
- 상품 클릭 → 상세 패널: 후기 10개, 평점, 원본 상품 페이지 링크.
- 각 카드 체크박스 → 상단 고정 바 "선택 N개 카드뉴스 만들기" 버튼.
- 버튼 동작: `http://localhost:8787`(로컬 파이프라인 앱)으로 선택 목록 전송.
  - 로컬 앱 미기동 시: "PC에서 파이프라인 앱을 켠 뒤 다시 눌러 주세요" 안내 (조회 기능은 영향 없음).
- 진행 상황 표시: 문구 생성 중 → 렌더 중 → 미리보기 → 업로드 중 → 완료(게시 링크).
- 기존 카드뉴스 생성기 화면은 유지하되 별도 라우트/탭으로 이동 (대시보드가 메인).

### 4) pipeline/ — 로컬 원클릭 앱

FastAPI 앱, 포트 8787, 본인 PC에서만 실행. 사이트 버튼 → 아래를 순차 실행:

1. **수집**: 선택 상품 정보·후기를 Firestore에서 로드, 상품 이미지를 로컬로 다운로드.
2. **문구 생성**: Anthropic API 호출 → C안 템플릿 구조(표지·상품 카드 N장·CTA)의 문구 +
   인스타 캡션 생성. 캡션·문구 톤은 확정된 "깔끔·절제" 규칙(메모리 `insta-caption-voice`) 적용.
3. **렌더**: C안 HTML 템플릿(card-drafts의 `longskirt-insta.html` 계열을 데이터 주입형으로 정리)에
   문구·이미지 주입 → Playwright 헤드리스 Chromium으로 1080×1350 JPG 렌더 →
   `바탕화면\카드뉴스\YYYYMMDD 주제\1.jpg~N.jpg` + `caption.txt` 저장 (기존 저장 규칙 유지).
4. **미리보기 게이트**: 사이트(또는 로컬 앱이 여는 페이지)에 완성 JPG 표시 → 사용자가 "게시" 클릭.
5. **게시**: `scripts/upload_instagram.py`의 로직(imgbb 임시 호스팅 → 캐러셀 컨테이너 → 게시) 재사용.

- 비밀키(`ANTHROPIC_API_KEY`, IG 토큰, imgbb 키, Firebase 서비스 계정)는 전부 로컬 `.env`.
  저장소에는 `.env.example`만 (CLAUDE.md 규칙).

## 에러 처리

- **크롤러**: 몰 API 응답 스키마가 예상과 다르면 해당 몰/카테고리만 건너뛰고 이전 스냅샷 유지,
  실패 내역 로그. 전체 실패가 반복되면 워크플로 실패로 표면화.
- **파이프라인**: 단계별 실패 시 중간 산출물(문구 JSON, JPG) 보존 → 실패 단계부터 재시도 가능.
  인스타 게시가 실패해도 JPG·캡션은 폴더에 남아 수동 게시 가능. `selections` 상태 필드로 추적.
- **사이트**: 로컬 앱 미기동·네트워크 오류를 한국어 안내로 구분 표시.

## 테스트 전략 (CLAUDE.md 준수: 테스트에서 실제 외부 호출 금지)

- 크롤러: 실제 응답을 1회 채집해 저장한 샘플 JSON으로 파서 단위 테스트. 네트워크 호출은 목.
- 파이프라인: Anthropic 호출 목, 렌더 결과는 Playwright로 실제 1080×1350 크기 검증(로컬 리소스만),
  업로드는 Graph API 목.
- 프론트: `npm run build` 통과 + 핵심 컴포넌트 렌더 확인.
- CI: 기존 `ci.yml`에 crawler·pipeline 테스트 편입.
- 실제 크롤링·Claude 호출·인스타 게시는 사용자 키로 로컬/Actions 운영 환경에서만 수행.

## 구축 순서

| Phase | 내용 | 완료 기준 |
|---|---|---|
| 0 | 크롤링 가능성 검증 (Actions 러너에서 두 몰 API 응답 확인) | 두 몰 랭킹·후기 응답 샘플 확보 (차단 시 실행 위치 전환 결정) |
| 1 | 크롤러 + Firestore 스키마 + 매시간 워크플로 | Firestore에 실데이터 스냅샷 적재, 파서 테스트 통과 |
| 2 | 조회 사이트 (랭킹 비교 + 상세 + 선택 UI) | 배포된 사이트에서 실데이터 조회 가능 |
| 3 | 로컬 파이프라인 (문구→렌더→미리보기→게시) | 목 기반 테스트 통과 + 로컬 리허설로 JPG 산출 |
| 4 | 통합 리허설 → 실전 게시 1회 | 사이트 선택→게시까지 전 과정 성공 |

## 범위 밖 (YAGNI)

- 두 몰 간 동일 상품 자동 매칭(정확도 한계) — 나란히 비교로 대체.
- 후기 감성 분석·요약 — 카드뉴스 문구 생성 시 Claude가 후기 원문을 재료로 쓰는 것으로 충분.
- 로그인/계정 시스템 — 조작 권한은 "로컬 앱이 내 PC에만 있음"으로 해결.
- 클라우드 파이프라인(B안) — 필요해지면 별도 프로젝트로.
